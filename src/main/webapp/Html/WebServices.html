Guia sobre Web Services, SOAP, RESful

¿Qué son los servicios web?

Lo primero que debe de estar claro es que cuál es la definición de servicio web.

Los servicios web son aplicaciones autónomas, autodescriptivas y modulares que se pueden publicar, encontrar e invocar en la web y que tienen mecanismos estándar para comunicarse con otros tipos de software en la red. "La idea es que cualquier cosa puede ser un servicio web". "Un servicio web es esencialmente una capa de infraestructura que existe entre el modelo de componentes". Desde una perspectiva empresarial, los servicios web permiten a las organizaciones integrar sin problemas diferentes aplicaciones. [1]

Eficaz sin tener que preocuparse por cómo están construidos, dónde viven, qué sistema operativo están ejecutando o cómo acceden a ellos. Por esta razón, los servicios web se consideran integrados porque permiten crear una interfaz para acceder a las aplicaciones, independientemente de sus características de implementación. [1]

Antes de hacer eso la integración requiere la creación de módulos de software especializados, pero con los servicios web actuales, los marcos basados en XML se utilizan como punto de partida. [1]

Hay cuatro estándares para un servicio web simple: XML, SOAP, UDDI y WSDL, los cuales al trabajar juntos proporcionan la funcionalidad básica de "solicitud/respuesta". Se puede entregar usando servicios web simples. [1]

Proporcionar de manera efectiva al sitio web información como noticias, estadísticas y pronósticos del tiempo, mientras los servicios web complejos pueden involucrar transacciones más complejas entre múltiples partes, incluidos socios comerciales o proveedores, y se basan en estándares de comercio electrónico como ebXML. Si los comerciantes necesitan reabastecer sus tiendas, pueden solicitar un presupuesto en línea mediante un servicio web. A su vez, los proveedores pueden responder automáticamente a las solicitudes y enviar sus ofertas a la tienda en línea y la aplicación que inició la solicitud. [1]

Los informes pueden seleccionar automáticamente la oferta con la mejor relación costo-beneficio. [1]

Características de los Servicios Web

Las características de los servicios Web son: [2]

El servicio debe estar disponible en la web. Para esto necesitas usar protocolos de transporte estándar como HTTP y mensajes de código como un lenguaje estándar que cualquier cliente que quiera utilizar el servicio puede conocer.

El servicio debe incluir una descripción del mismo. Así uno la aplicación podrá averiguar cuál es la función del servicio web y para qué sirve. Su interfaz para que cualquiera pueda usarla automáticamente aplicación sin intervención del usuario.

Debe ser capaz de localizar. Tenemos que tener algún mecanismo. Encuentre un servicio web que realice una función específica. Es por eso queremos dejar que la aplicación encuentre los servicios que necesita. Automáticamente sin el conocimiento previo del usuario.

Arquitectura de los Servicios Web

Las aplicaciones web de hoy ya no son suficientes. El modelo de comercio electrónico actual no facilita la integración de las aplicaciones de Internet con el resto del software empresarial. Si una empresa quiere aprovechar al máximo Internet, su sitio web debe seguir evolucionando. Este es el trasfondo de la aparición de los servicios web.

Como hemos visto, un servicio web es un componente de software de una aplicación empresarial que permite a los usuarios compartir datos con otros programas modulares a través de Internet.

Son aplicaciones independientes de la plataforma que se pueden publicar, localizar e invocar fácilmente mediante protocolos web estándar como XML, SOAP, UDDI o WSDL. El objetivo final es crear un directorio de servicios web en línea fácil de encontrar y altamente confiable. [3]

Ventajas de los Servicios Web

  • Aseguran la interoperabilidad de las aplicaciones de software independientemente de sus capacidades o plataforma de instalación. [4]
  • Los servicios web promueven estándares y protocolos basados en texto que facilitan el acceso a su contenido y la comprensión de su funcionamiento. [4]
  • Facilitan la unión de diferentes empresas de diferentes geografías para brindar servicios integrados. [4]

Desventajas de los Servicios Web

  • No son tan avanzados como los estándares abiertos para computación distribuida como CORBA (Common Object Request Broker Architecture) cuando se trata de realizar transacciones. [4]
  • Compatible con otros modelos de computación distribuida como RMI (Invocación de Método Remoto), CORBA o DCOM (Modelo de Objeto de Componentes Distribuidos). Esta es una de las desventajas de usar el formato de texto. La brevedad y la eficiencia de procesamiento tampoco son objetivos de XML. [4]
  • Al depender de HTTP, pueden evitar las medidas de seguridad basadas en cortafuegos cuyas reglas intentan bloquear o inspeccionar la comunicación entre programas a ambos lados de la barrera. [4]

Servicios web-SOAP

¿Qué es SOAP?

SOAP pretende ser un nuevo protocolo para entornos distribuidos y descentralizados que utiliza las capacidades de Internet y XML para transferir información escrita entre nodos. SOAP originalmente significaba Protocolo simple de acceso a objetos, pero en la versión 1.2 se cambió a solo SOAP porque no usa objetos directamente. SOAP es esencialmente un paradigma de intercambio de mensajes sin estado y unidireccional entre nodos SOAP (desde un remitente SOAP hasta un receptor SOAP). [5]

SOAP se puede utilizar para crear interacciones más complejas, como solicitud/respuesta, solicitud/respuesta múltiple, etc., al combinar intercambios unidireccionales con la funcionalidad proporcionada por el protocolo de transporte y/o la información de la aplicación subyacente. (Don Box, 2000). [5]

SOAP es un protocolo liviano que es independiente de la plataforma, el transporte y el sistema operativo, ya que se basa en marcos probados con el tiempo, como el protocolo HTTP y el marcado de texto XML. SOAP es una recomendación del W3C, lo que significa que es un informe técnico que es el resultado final de un amplio consenso, tanto dentro como fuera del W3C, sobre una tecnología o política en particular. Hay dos tipos de solicitudes SOAP. [5]

La primera son las llamadas a procedimientos remotos (RPC) ha pedido, similares a otras arquitecturas distribuidas. Esto suele ser síncrono; el cliente envía un mensaje y espera recibir una respuesta o un mensaje de error del servidor. El segundo tipo de solicitud SOAP es una solicitud de documento. En este caso, el cliente y el servidor envían/envían un documento XML completo en un mensaje SOAP. [5]

La estructura de SOAP

SOAP se basa en el metalenguaje XML. Esta es también una recomendación del W3C y es el conjunto de elementos de información necesarios para generar un documento XML bien formado (es decir, conforme a la estructura recomendada). SOAP absorbe su propia estructura de mensajes y, por lo tanto, se ajusta principalmente a los documentos XML. [6]

En la mayoría de los casos, SOAP también está integrado con HTTP. El transporte es realizado por el protocolo e integrado en su estructura. Un mensaje HTTP con una solicitud SOAP tiene la siguiente forma:


                            POST /example HTTP/1.1
                            Host: example.org
                            Content-Type: text/xml; charset=utf-8
                            <?xml version="1.0"? >
                            <SOAP-ENV:Envelope
                            xmlns:SOAP-ENV="http://www.w3.org/2003/05/soap-envelope"
                            SOAP-ENV:encodingStyle="http://www.w3.org/2001/12/soap-encoding">
                                    < SOAP-ENV:Header>
                                    < /SOAP-ENV:Header>
                                    < SOAP-ENV:Body>
                                    < /SOAP-ENV:Body>
                                    < /SOAP_ENV:Envelope>

En este ejemplo, la solicitud comienza con encabezados HTTP. El siguiente es el llamado sobre SOAP, que envuelve el contenido del mensaje real en un sobre. Los elementos principales de SOAP son el encabezado y el cuerpo. [6]

Características

  • Las operaciones son definidas como puertos WSDL. [7]
  • Dirección única para todas las operaciones.
  • Múltiples instancias del proceso comparten la misma operación.
  • Componentes fuertemente acoplados.

Ventajas

  • Fácil (generalmente) de utilizar. [7]
  • La depuración es posible.
  • Las operaciones complejas pueden ser escondidas detrás de una fachada.
  • Envolver API’s existentes es sencillo. Incrementa la privacidad.
  • Herramientas de desarrollo.

Desventajas

  • Los clientes necesitan saber las operaciones y su semántica antes de su utilización. [7]
  • Los clientes necesitan puertos dedicados para diferentes tipos de notificaciones.
  • Las instancias del proceso son creadas implícitamente.

Servicios web-RESTful

Estos dos conceptos clave son necesarios porque los servicios web RESTful son servicios web basados en la arquitectura REST. Los servicios web RESTful están basados en recursos. Un recurso es una entidad que se almacena principalmente en el servidor y un cliente solicita el recurso a través de servicios web RESTful. [8]

El uso de servicios web RESTful mejora la portabilidad entre cualquier tipo de plataforma. Esto permite a los desarrolladores ampliar fácilmente las aplicaciones. La principal ventaja es la separación entre cliente y servidor, ya que no importa de qué lenguaje o tipo de servidor provenga (Python, Java, Node.js); todo lo que se requiere es generar cada solicitud correctamente. [8]


Servicios Web RESTful
Operacion Metodo o HTTP Uri Parametros Resultados
Listar GET /{RECURSO} NO APLICA LISTAR TIPO DE RECURSO
Crear POST /{RECURSO} DENTRO DEL CUERPO SE CREA UN NUEVO RECURSO
Leer GET /{RECURSO}/{RECURSO_ID} NO APLICA RECURSO EN FUNCION(ID)
Actualizar PUT/PATCH /{RECURSO}/{RECURSO_ID} SE PASAN USANDO UNA CADENA SE ACTUALIZA/REEMPLAZA EL RECURSO
Borrar DELETE /{RECURSO}/{RECURSO_ID} NO APLICA SE ELIMINA EL RECURSO EN FUNCION(ID)

Características

  • Tiene cinco operaciones típicas: listar, crear, leer, actualizar y borrar. [8]
  • Cada operación requiere de dos cosas: El método URI y HTTP.
  • El URI es un sustantivo que contiene el nombre del recurso.
  • El método HTTP es un verbo.

Ventajas

  • Separacion cliente/servidor. [9]
  • Independencias de tecnologías/lenguajes.
  • Fiabilidad, escalabilidad, flexibilidad.
  • Experiencia de usuario.
  • Rest requiere menos recursos del servidor.

Desventajas

  • No tenemos estado en la API REST, lo que significa que debe configurar su propia infraestructura para mantener toda la aplicación. Por lo general, enviará un token al servidor que le indicará quién es usted y qué ha hecho en su aplicación. [9]
  • Inicialmente, es posible que necesite más tiempo de desarrollo porque debe configurar todo el sistema API. Pero en el futuro, especialmente si planea usar API de otros sistemas, puede recuperar el tiempo extra empleado y obtener mucha velocidad, porque el núcleo (API) del nuevo sistema ya está desarrollado. De cualquier manera, tendrá menos mantenimiento gracias a la separación de REST y código. [9]
  • En algunos casos, puede haber más rigor en el desarrollo, especialmente porque se trata de dos proyectos separados y su backend está basado en REST, y los frontends pueden no estar sincronizados. Por ejemplo, encuentra en el lado del cliente que necesita contenido nuevo de la API, y las personas responsables de crearlo en el lado del servidor pueden desarrollarlo a un ritmo diferente y tomarse el tiempo para desarrollarlo. [9]
  • Requiere más conocimiento y te obliga a salir de la zona de confort en la que podríamos estar. Debe conocer no solo el idioma, la base de datos, la biblioteca, sino también el protocolo HTTP. [9]
